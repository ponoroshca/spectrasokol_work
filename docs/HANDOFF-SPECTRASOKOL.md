# SpectraSokol — журнал работ и передача проекта

Shopbot (Telegram VPN-магазин + веб-админка + мини-апп + планировщик) на базе Remnawave,
развёрнутый и добренный под бренд **SpectraSokol**. Этот документ — что сделано, как устроено
и на что смотреть при продолжении.

> **Секреты не здесь.** Токены ботов/платёжек, API-токен Remnawave, `SECRET_KEY`, IP серверов,
> дампы БД и бэкапы **намеренно не в репозитории** (см. `.gitignore`). Всё оперативное — в
> приватной памяти оператора и на самих серверах. Здесь — только код и процедуры.

---

## 1. Архитектура (по ролям, без адресов)

- **shopbot-сервер** — контейнер `remnawave-shopbot` (`python -m shop_bot`): бот aiogram +
  веб-админка Flask + мини-апп FastAPI + планировщик, в одном процессе. Тут же nginx (лендинг,
  кабинет, админка, заглушка) и SSL.
- **Remnawave-панель** — на отдельном сервере (VPN-панель + Postgres + Caddy). Shopbot ходит к
  ней по HTTP API (адрес и токен — в настройках админки, не в коде).
- **Мосты (РФ)** и **загран-ноды** — ноды Remnawave. Мосты решают проблему РФ-DPI (Reality-вход
  от клиента + Reality-outbound на загран-ноду). Подписка клиентов указывает на домен мостов
  (round-robin DNS, серое облако Cloudflare).

### Домены
- `spectrasokol.online` — основной лендинг (нейтральный домен для рекламы).
- `vpn-vless.ru` — **заглушка-обманка** (безобидный сайт, чтобы домен с «vpn» в имени не
  привлекал внимания РКН). Реальный сервис тут не отдаётся.
- `app.<домен>` — личный кабинет (мини-апп).
- `admin.<домен>` — веб-админка + вебхуки платежей.

Все настройки (токены, платёжки, тарифы, тексты, домен) — в **админке**, не в коде. БД создаётся
пустой при первом старте.

## 2. Что настроено в проде

- **Боты** запущены (основной + поддержка), автостарт включён (ключ `auto_start_bot` в таблице
  `other`, НЕ в `bot_settings`).
- **Remnawave** подключена: выдача/продление ключей проверены боем.
- **Миграция из SHM** выполнена одноразовым скриптом (юзеры/ключи/платежи/промокоды из MySQL-дампа
  в SQLite). Скрипт импорта — оперативный, в репо не входит.
- **Тарифы**: матрица «устройства × срок» (2/5/6/8/10 устройств × 1/3/6/12 мес), слайдер устройств
  в мини-аппе (`device_mode='tiers'` у хоста; базовое число — `base_device_<host>`; надбавки —
  `device_tiers`).
- **Платёжки**: YooKassa + CryptoBot. Вебхуки: `/yookassa-webhook`, `/cryptobot-webhook`.
- **Поддержка**: тикеты создаются форум-темами в приватной группе (`support_forum_chat_id`,
  бот-админ с правом Manage Topics).

## 3. Правки в коде (эта сессия)

### Брендинг
- Полный ребренд Alma → **SpectraSokol** во всех текстах/ассетах/дефолтах (кроме `almalinux` —
  это название ОС в спидтесте, не бренд). Логотип (сокол в щите) + перекраска мини-аппа из
  бирюзовой палитры в фиолетовую (`primary #1fc6d8 → #a855f7` и т.д.). Чужие FAQ-ссылки убраны.
- Иконки PWA/логотипы перегенерированы из нового лого.

### Лендинги (`landing/`)
- `index.html` — продающий лендинг (редизайн по образцу структуры maximavps.click, но в своём
  бренде и **адаптивно**): орбита-герой со странами, бегущая строка, 3 шага, тарифы с рабочим
  JS-слайдером устройств, статистика, преимущества, FAQ, финальный CTA. **Упор на кабинет,
  Telegram-кнопок нет.** Отдаётся на `spectrasokol.online` и (при желании) на корне.
- `decoy.html` — **заглушка от РКН**: безобидный личный блог «Медленное утро» (чай/книги),
  светлая тема, ноль намёков на VPN. Ставится на `vpn-vless.ru`.

### Согласие с документами перед оплатой
- В мини-аппе на экранах покупки и продления — чекбокс «Я принимаю Условия использования и
  Политику конфиденциальности» над кнопкой оплаты; кнопка активна только при отмеченной галочке
  (`app.html`: `_payIntent`/`_termsOk`, `setPaymentButtonState` гейтит на чекбокс; ссылки через
  `openLinkSafe`). URL документов — настройки `terms_url`/`privacy_url`.

### Кнопки/навигация бота
- Кнопка «Поддержка» в главном меню и в welcome — прямая ссылка на саппорт-бота (было — веб-апп).

### Рассылки и уведомления (админка → Уведомления)
- Тест-рассылка и алерт «Новая оплата» теперь идут **всем админам** (`get_admin_ids()`), а не
  одному (`admin_telegram_id`).
- В «Тестовом режиме» рассылки добавлено поле **«ID для тестовой отправки»** (можно несколько; пусто
  → всем админам). Предпросмотр рассылки тоже шлётся всем админам.
- **GIF→MP4**: при загрузке медиа в рассылку GIF автоматически конвертируется в MP4 (`ffmpeg` в
  Dockerfile), файл именуется `*.anim.mp4` → отправляется как анимация (иначе Telegram шлёт GIF
  документом, а `.mp4` — видео). См. `webhook_server/modules/other.py`.

### История платежей / устройства
- В истории платежей и в алерте «Новая оплата» название тарифа показывает **реальное число
  устройств** из `tier_device_count` (сетка тарифов — базовые планы, реальные устройства в
  метаданных). `database.py: get_paginated_transactions`, `bot/handlers.py: notify_admin_of_purchase`.

### Инфраструктура
- `Dockerfile` — добавлен `ffmpeg` (для GIF→MP4).

## 4. Эксплуатация — важные нюансы (грабли)

- **Деплой правок:** `.py`/шаблоны — залить файл + `docker compose restart`. `app.html` читается
  per-request, но при правке шаблонов Flask/handlers нужен рестарт. Длинные сборки на сервере —
  через `nohup`, не в foreground SSH (обрыв убьёт сборку).
- **Автостарт ботов:** ключ `auto_start_bot` — в таблице `other`, не `bot_settings`.
- **Уведомления клиентам:** мастер-тумблеры `reengage_enabled`, `notif_sub_expiry_enabled`,
  `notif_trial_expiry_enabled`. После импорта клиентов их стоит выключать до анонса, иначе
  планировщик разошлёт напоминания незнакомым получателям.
- **Обновление панели/нод:** версии xray на всех узлах должны совпадать (иначе Reality-хопы
  мост↔нода рвутся). ⚠️ Ноды, поставленные скриптами DigneZzZ, **монтируют свой бинарник xray с
  хоста** — при обновлении образа ноды до новой версии нужно ЛИБО убрать это монтирование (нода
  возьмёт встроенный патченый xray Remnawave), ЛИБО обновить host-бинарник. Просто смена тега
  образа — ломает (xray не стартует). Обновлять по одной ноде, канареечно, с проверкой.
- **Бэкап Remnawave перед изменениями конфига** — обязательно (штатный `remnawave backup` +
  дамп БД).

### Лимит устройств (HWID)
- Наш код лимит на ключи ставит верно, но Remnawave **применяет** его только при
  `HWID_DEVICE_LIMIT_ENABLED=true` в `.env` панели (по умолчанию выключено — тогда лимит
  игнорируется). Считается только совместимыми клиент-приложениями (Happ и т.п.); «сырой»
  xray-конфиг HWID не шлёт.
- Политика «новые строго, старые не трогаем» реализуется грандфазером (поднять лимит текущим
  ключам до заведомо большого) + включением флага; код продолжает ставить реальный лимит новым
  ключам и при продлении.

## 5. Структура репозитория

```
src/shop_bot/
  bot/            — Telegram-бот (aiogram)
  webapp/         — мини-апп/кабинет (FastAPI) + app.html + login.html
  webhook_server/ — админка (Flask) + вебхуки платежей + шаблоны
  data_manager/   — БД (SQLite), схема, планировщик, уведомления, ре-энгейджмент
  modules/        — Remnawave API
  support_bot/    — бот поддержки
landing/          — лендинг (index.html), заглушка (decoy.html), логотип
docs/             — документация
Dockerfile, docker-compose.yml, install.sh, .env.example, pyproject.toml
```

Быстрый старт: `cp .env.example .env` → вписать `SHOPBOT_SECRET_KEY` → `docker compose up -d --build`.
Остальное (токены, платёжки, Remnawave, тарифы, бренд) — в админке после старта.
